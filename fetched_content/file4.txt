# Key Learnings from Donation App with Authentication

This document captures non-trivial and unexpected learnings discovered during the implementation of the authentication system for the donation app.

## 1. Session-Based Authentication with Cookies

### Critical: credentials: 'include'
**Learning**: Session cookies are NOT sent automatically with fetch requests by default. You must explicitly include `credentials: 'include'` in EVERY fetch request.

```typescript
// ❌ WRONG - Session cookie won't be sent
fetch('http://localhost:3001/api/auth/me')

// ✅ CORRECT - Session cookie will be sent
fetch('http://localhost:3001/api/auth/me', {
  credentials: 'include'
})
```

**Why this matters**: Without this, the backend can't identify the user, and all protected routes will fail with 401 Unauthorized, even if the user is logged in.

**Where it's needed**:
- Every API request that needs authentication
- Both GET and POST/PUT/DELETE requests
- Even requests that don't require auth but benefit from knowing the user

## 2. CORS Configuration for Credentials

### Critical: CORS credentials flag
**Learning**: When using `credentials: 'include'`, the backend MUST have `credentials: true` in CORS configuration, and `origin` cannot be a wildcard.

```typescript
// ❌ WRONG - Won't work with credentials
app.use(cors({
  origin: '*'  // Wildcard not allowed with credentials
}))

// ✅ CORRECT - Specific origin required
app.use(cors({
  origin: 'http://localhost:5173',
  credentials: true  // Critical!
}))
```

**Error you'll see if this is wrong**:
```
Access to fetch at 'http://localhost:3001/api/auth/login' from origin 'http://localhost:5173'
has been blocked by CORS policy: The value of the 'Access-Control-Allow-Credentials' header
in the response is '' which must be 'true' when the request's credentials mode is 'include'.
```

## 3. SQLite ALTER TABLE Limitations

### Unexpected: Can't add foreign keys with ALTER TABLE
**Learning**: SQLite's `ALTER TABLE` doesn't support adding foreign key constraints directly. You must recreate the table.

```sql
-- ❌ DOESN'T WORK in SQLite
ALTER TABLE donations ADD COLUMN user_id INTEGER REFERENCES users(id);

-- ✅ WORKAROUND - Add column without constraint, or recreate table
ALTER TABLE donations ADD COLUMN user_id INTEGER;
-- Then handle FK constraint in CREATE TABLE for new databases
```

**Solution implemented**:
- Check if table exists and has the column
- If table doesn't exist, create with foreign key
- If table exists without column, ALTER TABLE to add column (without FK constraint)
- The constraint is enforced in new table creations

## 4. Express Session with TypeScript

### Tricky: Extending session interface
**Learning**: To add custom properties to `req.session`, you must use TypeScript's module augmentation.

```typescript
// src/types/express-session.d.ts
import 'express-session';

declare module 'express-session' {
  interface SessionData {
    userId: number;      // Now req.session.userId exists!
    username: string;
    role: 'admin' | 'client';
  }
}
```

**Without this**: TypeScript will error on `req.session.userId` saying "Property 'userId' does not exist".

**Critical detail**: The file must be in `src/types/` and referenced in `tsconfig.json` for TypeScript to pick it up.

## 5. Multer Filename Callback

### Unexpected: req.session accessible in multer config
**Learning**: The multer storage configuration functions have access to `req.session` because they run AFTER session middleware.

```typescript
const storage = multer.diskStorage({
  filename: (req, file, cb) => {
    const userId = req.session.userId;  // This works!
    cb(null, `user_${userId}${path.extname(file.originalname)}`);
  }
});
```

**Why this is useful**: You can use session data to generate unique, user-specific filenames without passing userId as a query parameter.

**Gotcha**: If the route doesn't have `requireAuth` middleware first, `req.session.userId` might be undefined.

## 6. React Context Pattern for Auth

### Best Practice: Separate user state from loading state
**Learning**: Having a separate `loading` state is critical to prevent flash of redirect.

```typescript
// ❌ BAD - Causes flash of login page
const { user } = useAuth();
if (!user) return <Navigate to="/login" />;

// ✅ GOOD - Waits for auth check to complete
const { user, loading } = useAuth();
if (loading) return <div>Loading...</div>;
if (!user) return <Navigate to="/login" />;
```

**Why**: On app load, `user` is initially `null` while we check if session exists. Without `loading` state, user sees login page for a split second before being redirected to intended page.

## 7. Bcrypt is Asynchronous

### Important: Always await bcrypt operations
**Learning**: Bcrypt's `hash()` and `compare()` return Promises. Using them synchronously will fail silently or cause runtime errors.

```typescript
// ❌ WRONG - Won't work
const hash = bcrypt.hash(password, 12);  // Returns Promise, not hash!

// ✅ CORRECT - Must await
const hash = await bcrypt.hash(password, 12);
```

**In functions**: Any function using bcrypt must be `async`:
```typescript
export const createUser = async (username, email, password) => {
  const password_hash = await bcrypt.hash(password, 12);
  // ...
}
```

## 8. FileReader for Image Preview

### Useful: FileReader creates data URLs for preview
**Learning**: To show an image preview before upload, use FileReader to convert File to data URL.

```typescript
const reader = new FileReader();
reader.onloadend = () => {
  setPreview(reader.result as string);  // Base64 data URL
};
reader.readAsDataURL(file);
```

**Then use it**: `<img src={preview} />` to show the preview.

**Why not just use File object**: You can't directly use a File object as an img src. You need either a data URL (for preview) or an object URL (`URL.createObjectURL(file)`).

## 9. FormData for File Uploads

### Critical: Don't set Content-Type header with FormData
**Learning**: When sending FormData, do NOT set the `Content-Type` header. The browser sets it automatically with the correct boundary.

```typescript
// ❌ WRONG - Will break file upload
fetch(url, {
  method: 'POST',
  headers: {
    'Content-Type': 'multipart/form-data'  // NO!
  },
  body: formData
})

// ✅ CORRECT - Let browser set Content-Type
fetch(url, {
  method: 'POST',
  // No headers!
  body: formData
})
```

**Why**: The browser needs to add a boundary parameter to Content-Type (e.g., `multipart/form-data; boundary=----WebKitFormBoundary...`). If you set it manually, the boundary is missing and upload fails.

## 10. React Router Navigate vs Redirect

### Modern: Use Navigate component, not Redirect
**Learning**: In React Router v6+, `<Redirect>` is removed. Use `<Navigate>` instead.

```typescript
// ❌ OLD (React Router v5)
import { Redirect } from 'react-router-dom';
if (!user) return <Redirect to="/login" />;

// ✅ NEW (React Router v6+)
import { Navigate } from 'react-router-dom';
if (!user) return <Navigate to="/login" replace />;
```

**The `replace` prop**: Use `replace` to replace current history entry (user can't go back to protected page after redirect).

## 11. SQLite ON DELETE Actions

### Useful: ON DELETE SET NULL preserves donation history
**Learning**: Using `ON DELETE SET NULL` for the foreign key keeps donation records when users delete their accounts.

```sql
CREATE TABLE donations (
  user_id INTEGER,
  FOREIGN KEY (user_id) REFERENCES users(id) ON DELETE SET NULL
);
```

**Result**: When user deletes account, their donations remain but with `user_id = NULL` (anonymous donations).

**Alternative options**:
- `ON DELETE CASCADE`: Deletes donations when user deleted (data loss!)
- `ON DELETE RESTRICT`: Prevents user deletion if they have donations
- `ON DELETE SET NULL`: Our choice - preserves donation data

## 12. Express Session Store Configuration

### Important: Better-sqlite3-session-store wrapping
**Learning**: The session store requires a specific wrapping pattern with better-sqlite3.

```typescript
import SqliteStore from 'better-sqlite3-session-store';
import session from 'express-session';

// ✅ CORRECT - Pass session to SqliteStore
app.use(session({
  store: new (SqliteStore(session))({
    client: db,
    // ...
  })
}))

// ❌ WRONG - Missing session parameter
app.use(session({
  store: new SqliteStore({ client: db })  // Won't work!
}))
```

**Why**: SqliteStore needs access to the session module to properly integrate with express-session.

## 13. Password Validation: Client vs Server

### Best Practice: Validate on both sides, but differently
**Learning**: Client-side validation provides immediate feedback, but server-side validation is the security boundary.

**Client-side**: Check format immediately for UX
```typescript
if (password.length < 8) {
  setError('Password must be at least 8 characters');
  return;  // Don't even send request
}
```

**Server-side**: Enforce requirements as security boundary
```typescript
if (!/[A-Z]/.test(password) || !/[a-z]/.test(password) || !/[0-9]/.test(password)) {
  return res.status(400).json({ error: 'Password too weak' });
}
```

**Why both**: Client-side can be bypassed (browser tools, Postman), so server MUST validate. But client-side improves UX by catching errors before network round-trip.

## 14. React useEffect Dependencies

### Tricky: user object as dependency
**Learning**: When `user` is in useEffect dependencies, the effect runs whenever user object changes (login, logout, profile update).

```typescript
useEffect(() => {
  if (user) {
    setName(user.username);
    setEmail(user.email);
  }
}, [user]);  // Runs when user logs in, logs out, or updates profile
```

**Gotcha**: If you update `user` object frequently, this effect will run frequently. In our case, it's exactly what we want (auto-fill form when user logs in).

## 15. SQLite Integer vs Real for IDs

### Important: lastInsertRowid is bigint in better-sqlite3
**Learning**: SQLite's `lastInsertRowid` returns a bigint, not a regular number, which can cause TypeScript issues.

```typescript
const info = stmt.run(/* ... */);
const id = info.lastInsertRowid as number;  // Cast to number for TypeScript
```

**Why cast**: Our IDs won't exceed Number.MAX_SAFE_INTEGER, so casting to number is safe and makes TypeScript happy.

**Alternative**: Keep it as bigint, but then all ID types must be bigint (more changes).

## 16. Express Middleware Order Matters

### Critical: Session middleware before auth routes
**Learning**: Middleware order is crucial. Session middleware must run before any routes that need session data.

```typescript
// ✅ CORRECT ORDER
app.use(session({ /* ... */ }));  // 1. Set up session
app.post('/api/auth/login', handler);  // 2. Route can access req.session

// ❌ WRONG ORDER
app.post('/api/auth/login', handler);  // req.session undefined!
app.use(session({ /* ... */ }));  // Too late
```

**Our middleware order**:
1. CORS (allow cross-origin requests)
2. express.json() (parse JSON body)
3. express-session (session management)
4. Static files (serve uploads)
5. Routes (handle requests)

## 17. React Conditional Rendering Patterns

### Pattern: Nested ternaries vs early returns
**Learning**: For complex conditions, early returns are clearer than nested ternaries.

```typescript
// ❌ HARD TO READ
return loading ? <Loading /> : user ? <Navigate to="/" /> : <LoginForm />;

// ✅ CLEAR
if (loading) return <Loading />;
if (user) return <Navigate to="/" />;
return <LoginForm />;
```

**Used in**: ProtectedRoute component for clean authentication check flow.

## 18. Better-SQLite3-Session-Store Column Naming

### Critical: Session table column must be named 'expire' not 'expired'
**Learning**: The `better-sqlite3-session-store` library has an undocumented requirement: the expiration column MUST be named `expire` (not `expired`).

**Error you'll see**:
```
SqliteError: no such column: expire
    at Database.prepare
    at SqliteStore.get
```

**Wrong schema**:
```sql
CREATE TABLE sessions (
  sid TEXT PRIMARY KEY,
  sess TEXT NOT NULL,
  expired INTEGER NOT NULL  -- ❌ WRONG - causes error
);
```

**Correct schema**:
```sql
CREATE TABLE sessions (
  sid TEXT PRIMARY KEY,
  sess TEXT NOT NULL,
  expire INTEGER NOT NULL   -- ✅ CORRECT - must be 'expire'
);
```

**Why this happens**:
- The library's source code specifically looks for a column named `expire`
- This is not documented in the library's README
- The error only appears when trying to use sessions (e.g., login)

**How we discovered it**:
- Login was returning "Unexpected token '<', "<!DOCTYPE "... is not valid JSON"
- This meant an HTML error page was being returned instead of JSON
- Backend logs showed: `SqliteError: no such column: expire`
- Reading the library source code revealed it hardcodes the column name

**Fix**:
1. Drop the sessions table or delete the database
2. Recreate with correct column name
3. Restart the backend server

## 19. TypeScript Declaration Files for Untyped Packages

### Critical: Creating .d.ts files for packages without types
**Learning**: Some npm packages don't have TypeScript type definitions. You must create a declaration file manually.

**Problem encountered**: better-sqlite3-session-store doesn't have @types package
```
error TS7016: Could not find a declaration file for module 'better-sqlite3-session-store'
```

**Solution**: Create a type declaration file in `src/types/`

```typescript
// src/types/better-sqlite3-session-store.d.ts
declare module 'better-sqlite3-session-store' {
  import session from 'express-session';
  import Database from 'better-sqlite3';

  interface SqliteStoreOptions {
    client: Database.Database;
    expired?: {
      clear?: boolean;
      intervalMs?: number;
    };
  }

  function SqliteStore(session: typeof session): {
    new (options: SqliteStoreOptions): session.Store;
  };

  export = SqliteStore;
}
```

**Key points**:
- File must be in a location TypeScript scans (e.g., `src/types/`)
- Use `declare module` with exact package name
- Define interfaces for configuration options
- Match the actual runtime behavior of the package

**How to figure out types**:
1. Read the package's JavaScript source code
2. Check the package's documentation for parameters
3. Test in runtime to see what's actually passed/returned
4. Start with loose types (any) and refine as needed

## 20. React TypeScript: Type-Only Imports

### Modern: Use type-only imports for types with verbatimModuleSyntax
**Learning**: TypeScript's `verbatimModuleSyntax` option requires separating type imports from value imports.

```typescript
// ❌ ERROR with verbatimModuleSyntax enabled
import React, { createContext, ReactNode } from 'react';

// ✅ CORRECT - Separate type imports
import React, { createContext } from 'react';
import type { ReactNode } from 'react';
```

**Error message**:
```
TS1484: 'ReactNode' is a type and must be imported using a type-only import
when 'verbatimModuleSyntax' is enabled.
```

**Why this matters**:
- `verbatimModuleSyntax` ensures cleaner JavaScript output
- Type imports are completely removed at compile time
- Value imports become actual JavaScript imports
- Helps with tree-shaking and bundle size

**When to use type-only imports**:
- Interface imports: `import type { MyInterface } from './types'`
- Type alias imports: `import type { MyType } from './types'`
- TypeScript-only imports: `import type { ReactNode, FC } from 'react'`

**When NOT to use type-only imports**:
- Function imports: `import { useState } from 'react'` (runtime value)
- Component imports: `import { Button } from './Button'` (runtime value)
- Class imports: `import { MyClass } from './MyClass'` (runtime value)

## 21. React Unused Import Warning

### Best Practice: Remove unused imports
**Learning**: In modern React (17+), you don't need to import React in every file if you're only using JSX.

```typescript
// ❌ NOT NEEDED (React 17+)
import React from 'react';
import { useState } from 'react';

function MyComponent() {
  return <div>Hello</div>;  // JSX works without React import
}

// ✅ CORRECT - Only import what you use
import { useState } from 'react';

function MyComponent() {
  return <div>Hello</div>;
}
```

**Why**: React 17+ introduced the new JSX transform that doesn't require React to be in scope.

**Exception**: Still need React import when using:
- `React.createContext()`
- `React.forwardRef()`
- `React.memo()`
- Other React namespace methods

**In our code**: We removed unnecessary React import from App.tsx because we only used routing components, not React namespace methods.

## Summary

The most critical learnings:
1. **Always use `credentials: 'include'`** with session-based auth
2. **CORS must allow credentials** and specify exact origin
3. **Bcrypt is async** - always await it
4. **Don't set Content-Type** with FormData
5. **Separate loading state** prevents flash of redirect
6. **Middleware order matters** in Express
7. **Session table column must be 'expire'** (not 'expired') for better-sqlite3-session-store
8. **Create .d.ts files** for untyped npm packages
9. **Use type-only imports** for TypeScript types with verbatimModuleSyntax

These patterns and gotchas would take hours to debug without prior knowledge!

## Debugging Tips Learned

### 1. Session Cookie Not Sent
**Symptoms**: Protected routes return 401 even after login
**Check**:
- `credentials: 'include'` in fetch
- `credentials: true` in CORS
- Cookie visible in browser DevTools → Application → Cookies

### 2. File Upload Fails
**Symptoms**: Multer doesn't receive file, req.file is undefined
**Check**:
- No `Content-Type` header set manually
- FormData key matches multer field name
- File size within limit
- File type allowed

### 3. TypeScript Compilation Errors
**Symptoms**: Build fails with module not found or type errors
**Check**:
- Declaration files exist in `src/types/`
- Type-only imports use `import type`
- Unused imports removed
- tsconfig.json includes types directory

### 4. React Router Not Working
**Symptoms**: Navigation doesn't work, 404 on refresh
**Check**:
- BrowserRouter wraps the app
- Routes component contains Route children
- Paths start with `/`
- Protected routes wrapped with ProtectedRoute

### 5. Database Migration Issues
**Symptoms**: Column doesn't exist, foreign key errors
**Check**:
- Migration code checks if table/column exists first
- Foreign keys work differently in SQLite (recreate table)
- Better-sqlite3 uses synchronous API (no await needed)

These debugging patterns saved hours during development!
